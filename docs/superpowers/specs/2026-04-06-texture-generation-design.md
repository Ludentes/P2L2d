# Texture Generation Pipeline — Design Spec

**Date:** 2026-04-06
**Status:** Approved for implementation
**Author:** brainstorming session

---

## Goal

Given a single portrait photo, generate per-region textures that can be pasted into a
Live2D rig's texture atlas using the existing `swap_regions()` pipeline. Produce a dict
`{region_name: PIL.Image}` that feeds directly into `swap_regions(atlases, atlas_cfg, replacements)`.

---

## Goal Clarification: Identity Transfer With Rig-Constrained Style

The pipeline transfers a person's **appearance** (face, hair color, eye color) onto
the rig. Style transfer is a **rig/template constraint**, not a universal step:

- `humanoid-anime` template: portrait must be stylized to anime flat-cell art before
  pasting, because the atlas expects anime-style textures to blend with the rig's
  existing artwork (hair, clothing, accessories all drawn in anime style).
- `humanoid-realistic` (future): no style transfer — paste portrait pixels directly.

The template schema declares what style is required. The pipeline applies it.
This means stylization is configuration, not a hardcoded step.

## Background: The Multi-Layer / See-Through Problem

Live2D rigs are layered: face skin is a base layer drawn first; eye/brow/mouth drawables
are drawn on top at render time; hair can occlude both. The texture atlas stores ALL
layers flat and independently — there is no compositing at texture-write time.

This means the face skin atlas region must contain **clean skin where the animated
features (eyes, brows, mouth) will appear** — not the portrait's eyes/brows/mouth,
which would double-render on top of the animated drawables. CartoonAlive (arXiv:2507.17327)
calls this "underlying face repainting" and makes it an explicit pipeline stage.

**ComfyUI-See-through** (arXiv:2602.03749, SIGGRAPH 2026) is noted as a relevant tool
but is designed to decompose *anime art* into layers — not photos. It is deferred:
useful only if the rig ever uses art generated by a generative pipeline rather than
a portrait-derived one.

## Architecture: Two Phases

Both phases share the same output contract and feed into `swap_regions()` unchanged.

### Phase 1 — Stylize + Warp + Inpaint

```
portrait.jpg
    │
    ├─► style_transfer.py  [ComfyUI — template-driven]
    │       load template's texture_generation config
    │       if style_transfer == "anime_flat_cell":
    │           FLUX Kontext img2img or NoobAI-XL img2img → anime-stylized portrait
    │       if style_transfer == "none":
    │           pass through unchanged
    │       output: stylized_portrait
    │
    ├─► face_align.py
    │       MediaPipe landmarks on stylized_portrait → affine warp to template space
    │       output: warped_portrait
    │
    ├─► [per-region crop, pure Python]
    │       face_skin, left_eye, right_eye, left_eyebrow, right_eyebrow,
    │       mouth, left_cheek, right_cheek — cropped from warped_portrait
    │
    ├─► face_inpaint.py  [ComfyUI / FLUX Fill]
    │       mask = union of eye + brow + mouth bboxes on face_skin crop
    │       inpaint masked area → clean skin matching surrounding texture
    │
    └─► hair_segment.py  [ComfyUI / BiRefNet]
            hair mask on stylized_portrait (RGBA, feathered)
            affine-warp hair mask to template space
            crop per hair region (hair_front, hair_back, hair_side_left, hair_side_right)
    │
    ▼
swap_regions(atlases, atlas_cfg, replacements)
    ▼
validate_textures(rig, modified_atlases)
```

### Phase 2 — Stylize + See-Through Decompose (better quality for anime rigs)

Once ComfyUI-See-through weights are confirmed publicly available:

```
portrait.jpg
    │
    ├─► style_transfer.py  [ComfyUI — same as Phase 1]
    │       → anime-stylized portrait (1024px+, flat cell-shaded)
    │
    ├─► layer_decompose.py  [ComfyUI / ComfyUI-See-through]
    │       → 19 semantic RGBA layers, each fully inpainted
    │       layers: face, front_hair, back_hair, eyes_L/R, brows_L/R,
    │               mouth, neck, clothing, etc.
    │       eliminates separate inpaint + BiRefNet steps
    │
    └─► layer_map.py
            map See-through layer names → atlas region names
    │
    ▼
swap_regions(atlases, atlas_cfg, replacements)  ← identical to Phase 1
```

Phase 2 prerequisite: verify See-through weights are public (paper conditionally
accepted SIGGRAPH 2026, arXiv:2602.03749). Phase 2 applies only to anime-style
templates — for `humanoid-realistic`, Phase 1 without style_transfer is sufficient.

## Template Schema Extension

Add `[texture_generation]` section to `templates/{name}/schema.toml`:

```toml
[texture_generation]
style_transfer = "anime_flat_cell"   # "none" | "anime_flat_cell"
style_model    = "noobai-xl"         # ComfyUI checkpoint name (ignored if "none")
style_strength = 0.65                # img2img denoise strength (0.0–1.0)
```

`humanoid-anime/schema.toml` → `style_transfer = "anime_flat_cell"`
`humanoid-realistic/schema.toml` (future) → `style_transfer = "none"`

The `texture_gen.py` orchestrator reads this config and conditionally runs
`style_transfer.py`. No code changes needed when adding new templates.

## Components

### `pipeline/style_transfer.py`

**What it does:** Stylizes a portrait to match the target rig's art style via ComfyUI.
For `humanoid-anime`: FLUX Kontext img2img (preserves identity better than
text-only FLUX) or NoobAI-XL img2img at configurable strength.

**Interface:**
```python
async def stylize_portrait(
    portrait: Image.Image,
    style: str,                 # from template schema: "anime_flat_cell" | "none"
    model: str,                 # ComfyUI checkpoint name
    strength: float,            # denoise strength
    client: ComfyUIClient,
) -> Image.Image:
    """Returns stylized portrait. Returns portrait unchanged if style == 'none'."""
```

ComfyUI workflow: `comfyui/workflows/style_transfer_anime.json`

### `pipeline/face_align.py`

**What it does:** Detects facial landmarks in a portrait using MediaPipe FaceLandmarker,
loads template landmarks (stored per-template at
`templates/{template_name}/face_landmarks.json`), computes an affine transform, and
returns the warped portrait as a PIL Image sized to the atlas template space.

**Interface:**
```python
def detect_landmarks(image: Image.Image) -> np.ndarray:
    """Returns (N, 2) float array of landmark pixel positions."""

def load_template_landmarks(template_name: str) -> np.ndarray:
    """Loads template/face_landmarks.json for the given template."""

def align_portrait(
    portrait: Image.Image,
    template_name: str,
    output_size: tuple[int, int] = (512, 512),
) -> Image.Image:
    """Detect landmarks, compute affine warp, return warped portrait."""
```

Template landmarks are generated once per template using
`scripts/generate_template_landmarks.py` (renders the rig at neutral pose, runs
MediaPipe, saves the result). They are version-controlled alongside the template.

### `pipeline/face_inpaint.py`

**What it does:** Takes the warped face crop and a pixel mask covering the eye/brow/mouth
regions, runs FLUX Fill inpainting via ComfyUI to replace those pixels with plausible
clean skin.

**Interface:**
```python
async def inpaint_face_skin(
    face_crop: Image.Image,
    mask: Image.Image,               # white = inpaint, black = keep
    client: ComfyUIClient,
    prompt: str = "smooth skin, same skin tone as surrounding area, no facial features",
) -> Image.Image:
```

ComfyUI workflow: `comfyui/workflows/face_inpaint.json`
Model: FLUX.1-Fill-dev (or FLUX.1-Fill-schnell for speed)

### `pipeline/hair_segment.py`

**What it does:** Segments hair from a portrait (or anime art) using BiRefNet via ComfyUI,
affine-warps the mask to template space, and crops per-region RGBA images for each hair
region in the atlas.

**Interface:**
```python
async def segment_hair(
    portrait: Image.Image,
    client: ComfyUIClient,
) -> Image.Image:
    """Returns RGBA image: hair pixels kept, everything else transparent."""

def extract_hair_regions(
    hair_rgba: Image.Image,
    atlas_cfg: AtlasConfig,
    region_names: list[str],
) -> dict[str, Image.Image]:
    """Crop per-region RGBA from aligned hair image."""
```

ComfyUI workflow: `comfyui/workflows/hair_segment.json`
Model: BiRefNet (general or anime variant)

### `pipeline/texture_gen.py`

**What it does:** Orchestrates the full Phase 1 pipeline. Entry point for the generation
system.

**Interface:**
```python
async def generate_textures(
    portrait: Image.Image,
    atlas_cfg: AtlasConfig,
    rig: RigConfig,
    client: ComfyUIClient,
    template_name: str = "humanoid-anime",
) -> dict[str, Image.Image]:
    """
    Returns {region_name: PIL.Image} ready for swap_regions().
    Covers: face_skin, left_eye, right_eye, left_eyebrow, right_eyebrow,
            mouth, hair_front, hair_back, hair_side_left, hair_side_right.
    Skips: body, clothing (hue_shift handled separately), cloth_and_body.
    """
```

### `pipeline/package.py`

**What it does:** Copies the rig's model files (moc3, model3.json) plus the modified
atlases into a self-contained output directory. The output dir is the deliverable that
`muse-vtuber` loads directly as a Live2D model.

**Interface:**
```python
def package_output(
    rig: RigConfig,
    modified_atlases: dict[int, Image.Image],
    output_dir: Path,
) -> Path:
    """
    Writes to output_dir:
      - character.model3.json  (copied from rig.model3_json_path)
      - character.moc3         (copied from rig.moc3_path)
      - textures/              (original atlases overwritten by modified_atlases)

    Returns output_dir.
    """
```

Does NOT copy `model.pt` — the MLP is template-level and lives in `templates/`;
`muse-vtuber` loads it from there via the manifest.

### `pipeline/run.py`

**What it does:** Top-level orchestrator and CLI entry point. Connects
`generate_textures()` → `swap_regions()` → `package_output()` into a single callable
and exposes it as `python -m pipeline.run`.

**Interface:**
```python
async def run_portrait_to_rig(
    portrait_path: Path,
    rig_config: RigConfig,
    atlas_cfg: AtlasConfig,
    output_dir: Path,
    template_name: str = "humanoid-anime",
    client: ComfyUIClient | None = None,
) -> Path:
    """
    Full texture pipeline: portrait → output artifact directory.

    Steps:
    1. generate_textures(portrait, atlas_cfg, rig_config, client, template_name)
       → replacements: dict[str, Image]
    2. load_atlases(rig_config) → atlases: dict[int, Image]
    3. swap_regions(atlases, atlas_cfg, replacements) → modified_atlases
    4. package_output(rig_config, modified_atlases, output_dir)

    Returns output_dir.
    """
```

**CLI:**
```
python -m pipeline.run portrait.jpg \
    --rig manifests/hiyori.toml \
    --atlas manifests/hiyori_atlas.toml \
    --out ./output/hiyori_portrait/ \
    [--template humanoid-anime]
```

`load_atlases(rig_config)` is a small helper in `pipeline/run.py` that opens each
texture file listed in `rig_config.textures` as a PIL Image, keyed by index.

### `scripts/generate_template_landmarks.py`

**What it does:** One-time script. Renders the rig at neutral pose using `RigRenderer`,
runs MediaPipe on the render, extracts landmark pixel positions, saves to
`templates/{template_name}/face_landmarks.json`. Must be re-run if the template rig
changes.

---

## Template Landmarks File Format

`templates/humanoid-anime/face_landmarks.json`:
```json
{
  "template": "humanoid-anime",
  "render_size": [512, 512],
  "generated_from": "hiyori",
  "landmarks": {
    "left_eye_center":    [x, y],
    "right_eye_center":   [x, y],
    "nose_tip":           [x, y],
    "mouth_center":       [x, y],
    "chin":               [x, y],
    "left_cheek":         [x, y],
    "right_cheek":        [x, y],
    "mediapipe_full":     [[x0, y0], [x1, y1], ...]
  }
}
```

The `mediapipe_full` array is the full 478-point landmark set for computing the affine
transform. Named landmarks are derived from it for diagnostics.

## Face Inpainting Mask Generation

CartoonAlive's approach (from arXiv:2507.17327, Stage 3): after pasting feature textures
into the atlas, it runs the feature-stripped rendering through the placement MLP to
predict each feature's position (x, y, scale), then **renders the rig at those predicted
positions to get binary masks** of the exact pixels each feature covers during animation.
Those pixels are then inpainted on the face_skin texture.

The key insight: the mask must cover not just the neutral-pose feature position but the
full range of pixels the feature CAN cover across all animation. The rig render at a
given parameter value gives exactly those pixels.

**Phase 1 simplification:** we use UV bbox masks instead of rig-rendered masks.
This is an approximation — the UV bbox overestimates the animated coverage (rectangular
rather than mesh-shaped) but is sufficient for Phase 1 and requires no MLP or extra
render pass.

```python
def build_face_inpaint_mask(
    atlas_cfg: AtlasConfig,
    face_region_name: str = "face_skin",
    feature_regions: list[str] = ["left_eye", "right_eye",
                                   "left_eyebrow", "right_eyebrow", "mouth"],
    dilation_px: int = 8,
) -> Image.Image:
```

For each feature region, compute its bbox relative to the face_skin bbox, create a
binary mask, dilate by `dilation_px`, return as a PIL Image (white = inpaint).

**Phase 2 enhancement:** replace bbox masks with rig-rendered masks. For each feature,
render the rig with that drawable at its predicted parameter range (min/max values),
read back the union of covered pixels, use as the inpainting mask. Requires the
placement MLP to predict feature positions first.

## ComfyUI Workflows

Three new workflow JSON files:

**`comfyui/workflows/style_transfer_anime.json`**
FLUX Kontext img2img (preferred) or NoobAI-XL img2img workflow. Inputs: image (uploaded),
strength (float), optional prompt. Output: stylized image. FLUX Kontext is preferred
because it preserves identity better than text-guided diffusion at the same denoise
strength.

**`comfyui/workflows/face_inpaint.json`**
FLUX.1-Fill inpainting workflow. Inputs: image (uploaded), mask (uploaded), prompt string.
Output: inpainted image. Parameters: steps=20, cfg=1.0 (FLUX is cfg-free), denoise=0.85.

**`comfyui/workflows/hair_segment.json`**
BiRefNet segmentation workflow. Input: image (uploaded). Output: RGBA image with hair
kept and background made transparent. No text prompt needed for BiRefNet.

## What Is NOT In Scope

- Body and clothing texture generation (hue_shift is sufficient for Hiyori dev; full
  per-region generation deferred to HaiMeng + proper atlas)
- `cloth_and_body` region — deferred; too ambiguous without pixel-level mask
- Phase 2 See-through decomposition — separate implementation after Phase 1 validates
  the pipeline end-to-end
- Style transfer for non-anime templates — `humanoid-realistic` (future) sets
  `style_transfer = "none"` in schema.toml and style_transfer.py passes through unchanged

---

## Error Handling

- `MediaPipeLandmarkError` if MediaPipe cannot detect face landmarks in portrait (no
  face found, or confidence too low). User must supply a different portrait.
- `ComfyUIConnectionError` (already exists in `comfyui/exceptions.py`) if ComfyUI
  is not running.
- If hair segmentation returns < 1% non-transparent pixels, warn and skip hair regions
  (may happen if portrait has no visible hair or extreme lighting).
